# iChat Pro 基于端到端加密的轻量级即时通讯系统需求文档

## 一、项目概述

### 1.1 项目名称

基于端到端加密的轻量级即时通讯系统

### 1.2 项目定位

本系统是一个参考 Telegram Desktop 交互形式设计的轻量级即时通讯系统。系统采用 Web 技术实现核心业务功能，并通过 Electron 对 Web 客户端进行桌面端封装，使用户能够以桌面应用的形式使用聊天系统。

系统重点实现用户注册登录、联系人管理、私聊通信、群聊通信、历史消息查看、管理员管理等基础即时通讯功能。在此基础上，系统在私聊和群聊模块中引入端到端加密机制：私聊消息由通信双方客户端完成加密与解密；群聊一期采用基础版逐成员加密方案，由发送者客户端分别为每个当前活跃群成员生成对应密文。服务端只负责用户认证、公钥管理、会话关系维护、密文消息存储和 WebSocket 转发，不保存用户私钥、会话密钥、文件密钥明文和聊天消息明文。

### 1.3 项目背景

随着即时通讯软件的普及，用户对通信效率、使用体验和隐私保护的要求不断提高。传统聊天系统通常由服务器保存和处理明文消息，一旦服务器受到攻击或数据库发生泄露，用户聊天内容可能面临较高的隐私风险。

本项目希望在课程大作业的范围内，设计并实现一个轻量级即时通讯系统。系统借鉴 Telegram Desktop 的三栏式界面布局，通过左侧会话列表、中间聊天窗口、右侧用户或群组信息面板，为用户提供直观的聊天体验。同时，系统通过端到端加密机制展示安全通信的基本思想，使私聊和群聊消息明文只在发送方与接收方客户端出现，服务端和数据库仅保存密文、随机数、认证标签、密钥版本、发送时间等必要元数据。

### 1.4 项目目标

本系统的主要目标如下：

1. 实现一个具备桌面客户端形态的轻量级即时通讯系统。
2. 支持用户注册、登录、退出和个人资料管理。
3. 支持联系人查找、好友申请、好友同意或拒绝等联系人管理功能。
4. 支持一对一私聊消息发送、接收和历史记录查看。
5. 支持私聊消息端到端加密，服务端只保存密文消息和必要元数据。
6. 支持群聊创建、群成员管理、群聊消息发送和群聊历史记录查看。
7. 支持群聊基础版逐成员端到端加密，使每个群成员接收属于自己的独立密文副本。
8. 支持管理员对用户、群组和消息记录进行基础管理，但管理员不能查看端到端加密消息明文。
9. 使用 SQLite 保存系统数据，并提供清晰的数据模型设计。
10. 使用 WebSocket 实现消息实时推送，提高聊天交互体验。
11. 通过 Electron 将 Web 系统封装为桌面客户端，增强项目展示效果。

### 1.5 技术约束

系统采用以下技术路线：

1. 后端框架：Django。
2. 页面渲染：Django Templates。
3. 局部刷新：HTMX。
4. 页面样式：Tailwind CSS。
5. 实时通信：Django Channels + WebSocket。
6. 数据库：SQLite。
7. 桌面端封装：Electron。
8. 私聊加密：浏览器端 Web Crypto API 或等价前端加密方案。

系统一期不采用完整前后端分离架构，不引入 Vue、React 等重型前端框架。Electron 只作为桌面端外壳，不承担核心业务逻辑。

## 二、系统用户与角色

### 2.1 普通用户

普通用户是系统的主要使用者。普通用户可以注册账号、登录系统、修改个人资料、添加联系人、发送私聊消息、创建或加入群聊、查看历史消息。

普通用户的主要需求包括：

1. 能够安全登录系统。
2. 能够方便地查找和添加联系人。
3. 能够与联系人进行一对一聊天。
4. 能够创建群聊并参与群组讨论。
5. 能够查看私聊和群聊历史消息。
6. 希望私聊和群聊内容不被服务器直接读取。
7. 当消息无法解密时，系统能够给出明确提示，而不是导致页面异常。

### 2.2 群主

群主是群聊的创建者，拥有群聊管理权限。群主除具备普通用户功能外，还可以修改群聊信息、邀请成员、移除成员和解散群聊。

群主的主要需求包括：

1. 创建群聊。
2. 邀请联系人加入群聊。
3. 移除群成员。
4. 修改群名称或群头像。
5. 解散自己创建的群聊。

### 2.3 系统管理员

系统管理员负责维护系统基本秩序和数据管理。管理员可以通过后台查看用户、群组和消息记录，对异常账号或违规内容进行处理。

管理员的主要需求包括：

1. 查看用户列表。
2. 禁用或恢复用户账号。
3. 查看群组列表。
4. 删除违规群组。
5. 删除违规群聊消息。
6. 查看基础系统数据和操作记录。

8. 大文件传输和云盘式文件管理。

## 四、业务需求分析

### 4.1 账号管理需求

系统应支持用户通过注册功能创建账号。用户注册时需要填写用户名、密码和昵称等基本信息。系统应检查用户名是否重复，并对密码进行安全存储。用户登录时，系统应验证账号和密码是否正确，并检查账号是否被禁用。

用户登录成功后进入聊天主界面。用户可以修改昵称、头像和个性签名等个人资料。

### 4.2 联系人管理需求

系统应支持用户根据用户名、昵称或用户 ID 查找其他用户。用户可以向目标用户发送好友申请。目标用户可以同意或拒绝申请。同意后，双方建立联系人关系，并可以发起私聊。

系统应支持用户查看联系人列表，也应支持删除已有联系人。

### 4.3 私聊通信需求

系统应支持两个联系人之间进行一对一私聊。发送方输入消息后，客户端先对消息内容进行加密，然后通过 WebSocket 将密文发送给服务器。服务器保存密文消息，并将密文转发给接收方。接收方客户端收到密文后，在本地完成解密并显示明文。

私聊消息应支持历史记录查看。用户打开私聊窗口时，系统从服务器获取历史密文消息，客户端解密后展示。

### 4.4 端到端加密需求

系统应在私聊和群聊模块中引入端到端加密机制。用户首次登录或初始化密钥时，客户端通过 Web Crypto API 生成 ECDH 身份密钥对，并将公钥及公钥指纹上传服务器保存。私钥只保存在客户端本地，不应上传服务器。

发送私聊消息时，发送方客户端使用自己的 ECDH 私钥和接收方的 ECDH 公钥执行密钥协商（ECDH），计算出共享秘密（shared secret），再通过 HKDF 从共享秘密中派生出会话密钥（session_key）。客户端使用 AES-GCM 算法和 session_key 对消息明文进行加密，生成密文（ciphertext）、随机数（nonce）和认证标签（auth_tag）。服务器只保存密文、随机数、认证标签、加密算法标识、发送者、接收者和发送时间等数据，不保存 session_key 和消息明文。接收方客户端收到密文后，通过相同的 ECDH + HKDF 流程派生出相同的 session_key，在本地完成 AES-GCM 解密并展示明文。

群聊基础版采用逐成员加密方案：发送者在群聊中发送消息时，分别为每个当前活跃群成员执行 ECDH 密钥协商并独立加密同一条消息，每个群成员收到的是属于自己的独立密文副本。服务器保存一条群消息逻辑记录，并为每个接收者保存一条对应的密文记录，不保存群聊消息明文。

该设计的目标是减少服务端接触聊天明文的机会，提高用户私聊和群聊数据的隐私保护能力。

### 4.5 群聊通信需求

系统应支持用户创建群聊。创建者默认为群主。群主可以邀请联系人加入群聊，也可以移除群成员。群成员可以在群聊中发送消息，其他群成员能够实时收到消息。

系统一期群聊采用基础版逐成员端到端加密方案。发送者发送群消息时，客户端分别为每个当前活跃群成员执行 ECDH 密钥协商并独立加密同一条消息，服务器为每个接收者保存一份独立的密文记录。每个群成员只能解密属于自己的密文副本。新加入的成员只能解密加入之后的消息，退出或被移除的成员不再收到新的密文。Signal 协议级别的 Sender Key 群聊加密方案作为后续扩展方向。

### 4.6 历史消息需求

系统应支持用户查看私聊和群聊历史消息。历史消息应按照时间顺序展示，并支持分页加载。私聊历史消息从数据库读取后仍为密文，需要客户端解密后显示。群聊历史消息由服务端按当前成员查询属于该成员的密文副本，再由客户端解密后展示；服务端不得直接展示群聊明文。

### 4.7 桌面客户端需求

系统应通过 Electron 对 Web 聊天系统进行桌面端封装。用户启动桌面客户端后，可以进入登录页面并使用聊天功能。Electron 客户端应提供接近桌面聊天软件的窗口体验，包括固定窗口尺寸、聊天主界面展示和本地桌面应用入口。

Electron 不直接处理用户、消息和数据库等核心业务逻辑，核心业务仍由 Django 后端完成。

### 4.8 管理员后台需求

系统应提供管理员后台。管理员可以查看用户列表、群组列表和消息记录。管理员可以禁用异常用户、恢复用户账号、删除违规群组或删除违规群聊消息。

对于私聊端到端加密消息，管理员只能看到消息密文和元数据，不能查看私聊明文内容。

## 五、功能需求

### 5.1 用户注册

功能编号：FR-01

功能名称：用户注册

参与者：普通用户

功能描述：用户填写用户名、密码、确认密码和昵称后提交注册请求。系统验证信息合法性，若验证通过，则创建用户账号。

输入信息：用户名、密码、确认密码、昵称。

输出信息：注册成功提示或注册失败原因。

基本流程：

1. 用户进入注册页面。
2. 用户填写注册信息。
3. 系统校验用户名、密码和确认密码。
4. 系统判断用户名是否已存在。
5. 系统保存用户账号信息。
6. 系统提示注册成功。

异常流程：

1. 用户名为空，系统提示“用户名不能为空”。
2. 用户名已存在，系统提示“用户名已被占用”。
3. 两次密码不一致，系统提示“两次密码输入不一致”。
4. 密码格式不符合要求，系统提示“密码格式错误”。

### 5.2 用户登录

功能编号：FR-02

功能名称：用户登录

参与者：普通用户、管理员

功能描述：用户输入账号和密码后，系统验证身份信息。验证成功后进入聊天主界面，验证失败则提示错误原因。

输入信息：用户名、密码。

输出信息：登录成功状态或登录失败提示。

基本流程：

1. 用户进入登录页面。
2. 用户输入用户名和密码。
3. 系统校验账号是否存在。
4. 系统校验密码是否正确。
5. 系统判断账号状态是否正常。
6. 系统允许用户进入聊天主界面。
7. 客户端前端脚本（JS）检测本地 LocalStorage 中是否存在当前用户的非对称密钥对（公钥和私钥）。
8. 若本地无密钥对（如首次登录或清空浏览器缓存），系统将自动调用 Web Crypto API 生成一套新的非对称密钥对，将私钥保存在本地 LocalStorage 中，并将公钥上传至 Django 服务端，同时在界面上提示用户备份下载私钥（以便于后续更换设备时解密历史消息）。
9. 若本地已存在对应的密钥对，则跳过生成和公钥上传步骤。

异常流程：

1. 用户名不存在，系统提示“账号不存在”。
2. 密码错误，系统提示“用户名或密码错误”。
3. 账号被禁用，系统提示“账号已被禁用”。

### 5.3 修改个人资料

功能编号：FR-03

功能名称：修改个人资料

参与者：普通用户

功能描述：用户登录后可以修改昵称、头像和个性签名等资料。

输入信息：昵称、头像、个性签名。

输出信息：修改成功提示或失败原因。

基本流程：

1. 用户进入个人资料页面。
2. 系统展示当前资料。
3. 用户修改资料信息。
4. 系统校验输入内容。
5. 系统保存修改结果。
6. 系统提示修改成功。

### 5.4 查找联系人

功能编号：FR-04

功能名称：查找联系人

参与者：普通用户

功能描述：用户通过用户名、昵称或用户 ID 查找其他用户。

输入信息：查询关键字。

输出信息：用户查询结果列表。

基本流程：

1. 用户进入联系人查找页面。
2. 用户输入查询关键字。
3. 系统根据关键字查询用户。
4. 系统展示查询结果。
5. 用户选择目标用户并发送好友申请。

### 5.5 发送好友申请

功能编号：FR-05

功能名称：发送好友申请

参与者：普通用户

功能描述：用户向其他用户发送好友申请，对方可以同意或拒绝。

输入信息：目标用户 ID、申请备注。

输出信息：申请发送结果。

基本流程：

1. 用户选择目标用户。
2. 用户填写申请备注。
3. 系统检查双方是否已经是联系人。
4. 系统创建好友申请记录。
5. 系统通知目标用户。
6. 系统提示申请发送成功。

异常流程：

1. 双方已经是联系人，系统提示“该用户已在联系人列表中”。
2. 已存在未处理申请，系统提示“请勿重复发送申请”。
3. 目标用户不存在，系统提示“用户不存在”。

### 5.6 处理好友申请

功能编号：FR-06

功能名称：处理好友申请

参与者：普通用户

功能描述：用户可以查看收到的好友申请，并选择同意或拒绝。

输入信息：好友申请 ID、处理结果。

输出信息：处理成功提示。

基本流程：

1. 用户进入好友申请页面。
2. 系统展示待处理申请。
3. 用户选择同意或拒绝。
4. 若用户同意，系统建立联系人关系。
5. 系统更新申请状态。
6. 系统提示处理成功。

### 5.7 发送加密私聊消息

功能编号：FR-07

功能名称：发送加密私聊消息

参与者：普通用户

功能描述：用户向联系人发送私聊消息。消息在客户端加密后，通过 WebSocket 发送给服务器，服务器保存密文并转发给接收方。

输入信息：接收方 ID、消息明文。

输出信息：消息发送结果。

基本流程：

1. 用户打开私聊窗口。
2. 用户输入消息内容。
3. 客户端获取接收方公钥或会话加密信息。
4. 客户端对消息明文进行加密。
5. 客户端通过 WebSocket 发送密文消息。
6. 服务器保存密文消息。
7. 服务器将密文消息转发给接收方。
8. 发送方聊天窗口显示已发送消息。

异常流程：

1. 消息内容为空，系统提示“消息不能为空”。
2. 接收方不是联系人，系统提示“无权发送私聊消息”。
3. 加密失败，系统提示“消息加密失败，请重试”。
4. WebSocket 连接断开，系统提示“连接已断开，请稍后重试”。

### 5.8 接收并解密私聊消息

功能编号：FR-08

功能名称：接收并解密私聊消息

参与者：普通用户

功能描述：接收方客户端接收服务器转发的密文消息后，在本地完成 ECDH 密钥协商、HKDF 密钥派生和 AES-GCM 解密，并将明文展示在聊天窗口中。服务器不参与解密过程。

输入信息：密文消息、nonce、auth_tag、发送方公钥版本、接收方公钥版本、加密算法标识。

输出信息：解密后的消息内容或解密失败提示。

基本流程：

1. 客户端通过 WebSocket 接收包含【消息密文 ciphertext、随机数 nonce、认证标签 auth_tag、发送方公钥版本、接收方公钥版本、加密算法标识】的数据包。
2. 客户端从本地存储中读取当前用户的 ECDH 私钥。
3. 客户端获取发送方公钥，并检查公钥指纹或密钥版本是否与本地记录一致。
4. 客户端使用“当前用户私钥 + 发送方公钥”执行 ECDH，计算共享秘密 shared_secret。
5. 客户端使用 HKDF 从 shared_secret 中派生出会话密钥 session_key。
6. 客户端使用 session_key、nonce 和 auth_tag 对 ciphertext 执行 AES-GCM 解密。
7. 解密成功后，在聊天窗口中通过 HTMX 局部刷新或 JS 动态渲染显示消息明文。
8. 系统向后端发送确认请求，更新消息接收/阅读状态。

异常流程：

1. 本地私钥丢失（如用户清除浏览器缓存且未做备份）：
   1a. 客户端无法计算 shared_secret。
   1b. 界面上该条消息气泡显示为：`[无法解密：缺少本地解密私钥，请导入备份的私钥或初始化新密钥]`，不影响页面其他功能正常运转。
2. 密文损坏或传输数据不完整：
   2a. AES-GCM 解密校验失败。
   2b. 系统提示“消息内容损坏，解密失败”。
3. 密钥不匹配或联系人公钥发生变化：
   3a. 客户端派生出的 session_key 无法解密当前密文。
   3b. 系统提示“当前设备密钥与消息加密密钥不匹配”或“对方安全密钥已变更，请重新验证身份”。

### 5.11 发送加密群聊消息

功能编号：FR-11

功能名称：发送加密群聊消息

参与者：群成员

功能描述：群成员在群聊中发送消息。系统一期采用基础版逐成员端到端加密方案，发送者客户端分别为每个当前活跃群成员生成独立密文，服务器只保存群消息逻辑记录和各接收者对应的密文记录，并通过 WebSocket 推送给群成员。

输入信息：群 ID、消息明文、当前活跃群成员列表、公钥信息。

输出信息：发送成功状态或失败提示。

基本流程：

1. 用户进入群聊窗口。
2. 用户输入群聊消息。
3. 系统判断用户是否属于该群，且成员状态是否为 active。
4. 客户端向服务器请求当前 active 群成员列表及其公钥。
5. 客户端分别为每个当前 active 群成员执行 ECDH + HKDF，派生对应会话密钥。
6. 客户端分别使用 AES-GCM 加密同一条群消息，生成多个接收者密文副本。
7. 客户端将群消息密文包发送给服务器。
8. 服务器保存一条 `group_message` 群消息逻辑记录。
9. 服务器为每个接收者保存一条 `group_message_recipient` 密文记录。
10. 服务器通过 WebSocket 将对应密文推送给各群成员。
11. 群成员客户端收到属于自己的密文后，在本地解密并显示消息。

异常流程：

1. 用户不属于该群，系统提示“无权限发送群消息”。
2. 群聊已解散，系统提示“群聊不存在或已解散”。
3. 消息内容为空，系统提示“消息不能为空”。
4. 获取群成员公钥失败，系统提示“群成员密钥信息获取失败，请稍后重试”。
5. 加密失败，系统提示“群消息加密失败，请重试”。
6. 某成员已退出或被移除，系统不再为该成员生成新密文。

## 六、非功能需求

### 6.1 安全性需求

1. 用户密码不得以明文形式存储。
2. 用户登录后才能访问聊天主界面。
3. 普通用户不得访问管理员后台。
4. 私聊和群聊消息应在客户端加密后再发送给服务器。
5. 服务器和数据库不保存私聊、群聊消息明文。
6. 用户私钥或本地解密信息原则上不上传服务器。
7. 服务器不生成、不保存 session_key 和 file_key 明文。
8. 系统应防止用户查看不属于自己的会话、群聊、消息和文件。
9. 管理员不能查看端到端加密消息明文。
10. 系统应对重要操作进行权限校验。
11. 系统应记录管理员关键操作日志。
12. 当联系人公钥发生变化时，客户端应提示用户重新验证身份。

### 6.2 易用性需求

1. 系统界面应参考 Telegram Desktop 的三栏式布局。
2. 左侧应展示会话列表和搜索入口。
3. 中间区域应展示当前聊天内容和消息输入框。
4. 右侧可展示用户资料、群组信息或成员列表。
5. 发送方和接收方消息气泡应有明显区分。
6. 系统应对错误操作给出明确提示。
7. 常用功能入口应清晰可见。
8. 用户切换会话时，页面应尽量局部刷新，不频繁整页刷新。

### 6.3 性能需求

1. 消息发送后应能在较短时间内显示在聊天窗口中。
2. 历史消息应采用分页加载。
3. 联系人列表和会话列表加载时间应控制在可接受范围内。
4. 系统应支持课程演示规模下的多用户同时聊天。
5. 消息查询应为会话 ID、发送时间等字段建立索引，提高历史消息查询效率。

### 6.4 可维护性需求

1. 系统应按模块划分代码结构。
2. 用户模块、联系人模块、聊天模块、群组模块、密钥模块和管理模块应尽量解耦。
3. 数据库表字段命名应清晰。
4. 关键业务逻辑应添加必要注释。
5. 系统应便于后续扩展文件发送、消息搜索、Sender Key 群聊加密、多设备同步等功能。
6. Electron 客户端不应耦合过多业务逻辑，核心业务应集中在 Django 系统中。
7. 端到端加密相关逻辑应尽量封装在前端加密模块中，避免散落在普通页面逻辑中。

### 6.5 可靠性需求

1. WebSocket 连接断开时，系统应提示用户连接状态异常。
2. 消息发送失败时，系统应允许用户重新发送。
3. 解密失败时，系统应提示“无法解密该消息”，而不是导致页面崩溃。
4. 用户登录状态过期后，系统应引导用户重新登录。
5. 数据库操作失败时，系统应返回合理错误提示。

## 七、数据需求

### 7.1 用户数据

系统需要保存用户账号和资料信息，包括用户 ID、用户名、密码哈希、昵称、头像、个性签名、账号状态、创建时间和更新时间。

### 7.2 好友申请数据

系统需要保存好友申请记录，包括申请 ID、申请人 ID、接收人 ID、申请备注、申请状态、申请时间和处理时间。

### 7.3 联系人数据

系统需要保存用户之间的联系人关系，包括关系 ID、用户 ID、联系人 ID、关系状态和创建时间。用户与用户之间的好友关系为多对多关联，通过联系人表进行拆分分解，从而降低多对多关系复杂度，呼应数据库关联表设计。

### 7.4 会话数据

系统需要保存会话信息，包括会话 ID、会话类型、创建者 ID、最近消息时间和创建时间。会话类型包括私聊会话和群聊会话。

### 7.5 消息数据

系统需要保存消息记录，包括消息 ID、会话 ID、发送者 ID、接收者 ID、群组 ID、消息类型、消息状态和发送时间等数据。

对于私聊消息，数据库只保存密文内容和加密元数据，不保存明文内容。私聊消息相关字段包括：

- **消息密文（ciphertext）**：由发送方客户端使用 AES-GCM 加密后的消息内容。
- **随机数（nonce/IV）**：每次加密生成的随机数，用于保证同一明文在不同消息中产生不同密文。
- **认证标签（auth_tag）**：AES-GCM 生成的完整性校验标签，用于检测密文是否被篡改。
- **加密算法标识（algorithm）**：用于标明所采用的算法，如 AES-GCM。
- **发送者密钥版本和接收者密钥版本**：用于判断消息加密时使用的公钥版本。

对于群聊消息，系统一期采用基础版逐成员端到端加密方案。数据库中应保存一条群消息逻辑记录，并为每个当前活跃群成员保存一条对应的接收者密文记录。群聊消息不保存普通明文文本。

对于图片、文件、表情包等多媒体消息，本文档仅保留 Phase 2 数据模型扩展方向。Phase 1 WebSocket 核心链路只交付文本消息。后续实现多媒体消息时，消息表保存消息类型和关联文件 ID，文件内容应通过 `file_key` 加密后保存为密文文件或密文分块，数据库只保存加密文件路径、加密后的 file_key、nonce、auth_tag 和文件元数据。

### 7.6 用户公钥数据

系统需要保存用户公钥信息，包括公钥 ID、用户 ID、公钥内容、算法类型、是否启用、创建时间和更新时间。

用户私钥不保存在服务器数据库中。

### 7.7 群组数据

系统需要保存群组信息，包括群 ID、群名称、群头像、群主 ID、群状态、创建时间和更新时间。

### 7.8 群成员数据

系统需要保存群成员关系，包括群成员 ID、群 ID、用户 ID、群内角色、加入时间和成员状态。群内角色包括群主、管理员和普通成员。

用户和群组之间是多对多关系，需要通过群成员表进行拆分。

### 7.9 管理日志数据

系统需要保存管理员操作日志，包括日志 ID、管理员 ID、操作对象、操作类型、操作时间和操作说明。

## 九、核心业务流程

### 9.1 用户登录流程

用户打开系统客户端后进入登录页面，输入用户名和密码。系统校验账号是否存在、密码是否正确、账号状态是否正常。验证通过后，系统建立登录状态并跳转至聊天主界面。若验证失败，系统显示错误提示。

### 9.2 添加联系人流程

用户在联系人页面输入关键字查找其他用户。系统返回查询结果后，用户选择目标用户并发送好友申请。目标用户收到申请后可以同意或拒绝。若目标用户同意，系统建立联系人关系，双方可以进行私聊。

### 9.3 发送加密私聊消息流程

发送方打开私聊窗口并输入消息。客户端获取接收方公钥或会话加密信息，在本地将明文消息加密为密文。客户端通过 WebSocket 将密文消息发送给服务器。服务器保存密文并转发给接收方。接收方客户端收到密文后，在本地解密并显示明文。

### 9.4 创建群聊流程

用户点击创建群聊，填写群名称并选择初始成员。系统创建群组记录，将创建者设置为群主，并建立群成员关系。创建成功后，群成员可以在会话列表中看到该群聊。

### 9.5 群聊消息发送流程

群成员进入群聊窗口并输入消息。系统首先校验用户是否属于该群，且成员状态是否为 active。校验通过后，客户端获取当前 active 群成员列表及其公钥，分别为每个群成员执行 ECDH + HKDF 并使用 AES-GCM 加密同一条群聊消息。客户端将群消息密文包发送给服务器。服务器保存一条群消息逻辑记录，并为每个接收者保存一条独立密文记录，然后通过 WebSocket 将对应密文推送给在线群成员。群成员客户端收到属于自己的密文后，在本地解密并显示消息。

### 9.6 管理员处理违规内容流程

管理员登录后台后查看用户、群组或消息列表。管理员选择目标对象并执行禁用、恢复、删除等管理操作。系统更新对应数据，并记录管理员操作日志。

## 十、系统界面需求

### 10.1 登录注册界面

系统应提供简洁的登录和注册界面。用户能够在登录和注册之间切换。界面应展示系统名称、输入框、提交按钮和错误提示区域。

### 10.2 聊天主界面

聊天主界面采用三栏式布局：

1. 左侧栏：展示搜索框、会话列表、联系人入口和群聊入口。
2. 中间栏：展示当前会话的消息记录、消息输入框和发送按钮。
3. 右侧栏：展示联系人资料、群组信息、群成员列表和管理入口。

### 10.3 私聊窗口

私聊窗口应展示聊天对象信息、历史消息、消息输入框和发送按钮。私聊消息气泡应区分自己发送和对方发送的消息。若消息无法解密，应显示无法解密提示。

### 10.4 群聊窗口

群聊窗口应展示群名称、群成员消息、消息输入框和发送按钮。群主应能看到群管理入口。群聊窗口顶部应显示“端到端加密已开启”或类似安全提示。群成员变化时，消息区应显示系统提示，例如“群成员发生变化，后续消息将仅对当前成员加密”。若某条群消息无法解密，应在对应消息气泡中显示无法解密提示。

### 10.5 管理员后台界面

管理员后台应支持查看用户、群组和消息记录。管理员可以执行禁用用户、恢复用户、删除群组和删除消息等操作。

## 十二、验收标准

1. 用户能够完成注册、登录和退出。
2. 用户能够查找其他用户并发送好友申请。
3. 用户能够同意或拒绝好友申请。
4. 用户能够向联系人发送私聊消息。
5. 私聊消息能够通过 WebSocket 实时发送和接收。
6. 私聊消息在数据库中以密文形式保存。
7. 接收方能够在客户端解密并查看私聊消息。
8. 用户能够查看私聊历史消息。
9. 用户能够创建群聊并邀请成员。
10. 群成员能够发送和查看群聊消息。
11. 群聊消息在数据库中以群消息逻辑记录和接收者密文记录的形式保存。
12. 群成员只能拉取和解密属于自己的群聊密文副本。
13. 群主能够管理群成员。
14. 管理员能够登录后台并管理用户、群组和消息。
15. 管理员不能查看私聊或群聊消息明文，只能查看密文和元数据。
16. 系统能够通过 Electron 以桌面客户端形式运行或展示。
17. 系统能够对异常输入和异常操作给出合理提示。
18. 系统至少提供登录模块、私聊模块、群聊模块和端到端加密相关测试用例。
19. 系统数据库表结构、ER 图、类图和实际实现应保持一致。
