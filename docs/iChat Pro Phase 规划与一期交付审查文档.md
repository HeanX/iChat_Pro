# iChat Pro Phase 规划与一期交付审查文档

> 状态：Review v1.0
> 审查日期：2026-06-03
> 审查对象：需求文档、TODO、设计文档、当前代码、自动化测试与开放 GitHub Issues
> 结论：Phase 1 尚未达到可验收状态；完成现有开放 Issue 后仍需补充若干基础软件收口任务

## 1. 文档目的

本文档用于明确 iChat Pro 的阶段边界，并审查 Phase 1 是否覆盖基础即时通讯软件的完整交付范围。

项目采用两阶段路线：

| 阶段 | 定位 | 核心目标 |
| --- | --- | --- |
| Phase 1 | 基础软件功能 | 完成可运行、可演示、可验证的 E2EE 轻量即时通讯桌面软件 |
| Phase 2 | 扩展生态 | 增加 Channel、Bot、LLM Bot、Agent 和 OpenClaw Adapter |

Phase 2 不应阻塞 Phase 1，也不应在 Phase 1 尚未形成真实聊天闭环时提前进入实现。

------

## 2. Phase 1 范围

### 2.1 必须交付

Phase 1 只交付基础软件功能：

| 模块 | 必须交付能力 |
| --- | --- |
| 账号 | 注册、登录、退出、禁用账号拦截 |
| 资料 | 昵称、头像、个性签名编辑 |
| 联系人 | 按用户名、昵称或用户 ID 查找；发送、同意、拒绝好友申请；删除联系人 |
| 私聊 | 创建或获取私聊会话、真实会话列表、文本消息实时收发、历史分页、送达与已读状态 |
| 群聊 | 创建群聊、修改群信息、邀请和移除成员、成员退出、解散群聊、成员列表 |
| E2EE | 客户端密钥初始化、备份与恢复；私聊加密；群聊逐成员加密；密钥变化和损坏提示 |
| 实时通信 | 单一 `/ws/chat/` 连接、身份校验、重连、错误反馈、必要状态推送 |
| 管理后台 | 管理用户、群组和消息元数据；禁止查看消息明文；记录关键管理员操作 |
| 桌面端 | Electron 外壳加载稳定的 Django Web 页面 |
| 交付 | 自动化测试、手动联调记录、README、协议说明、数据模型说明和演示步骤 |

### 2.2 明确不做

以下能力不属于 Phase 1：

1. Channel。
2. Bot、LLM Bot 和 Agent。
3. OpenClaw Adapter。
4. Signal Protocol、Double Ratchet 和 Sender Key。
5. 完整前向保密。
6. 多端同步。
7. 移动端 App。
8. 语音和视频通话。
9. 图片、文件、表情包上传和大文件分块传输。
10. 开放式插件或小程序生态。

Phase 1 WebSocket 核心链路只要求文本消息。Twemoji 方案属于文本 Emoji 的展示一致性优化，不改变消息协议。

## 3. Phase 2 范围

Phase 2 在 Phase 1 通过验收后启动：

| 模块 | 扩展能力 |
| --- | --- |
| Channel | 公开或私有频道、订阅、管理员发布、Bot 草稿和发布 |
| Bot | 系统 Bot、命令 Bot、通知 Bot、Channel 发布角色 |
| LLM Bot | 问答、翻译、摘要、知识库检索、草稿生成 |
| Agent | OpenClaw 等独立 Agent Gateway、受限工具调用、人工审批 |
| 安全治理 | Gateway 隔离、能力白名单、预算限制、审计日志、Prompt Injection 防护 |
| 高安全扩展 | Bot 作为 E2EE 成员、Electron 本地 Agent、Sender Key 评估 |

详细设计见：

```text
docs/iChat Pro Bot、LLM Agent 与 Channel 扩展方案文档.md
```

## 7. 尚未被完整覆盖的 Phase 1 缺口

### 7.1 [T26：补齐联系人搜索口径](https://github.com/HeanX/iChat_Pro/issues/36)

需求文档要求按用户名、昵称或用户 ID 查找联系人。当前实现仅查询：

```text
username__icontains
```

需要：

1. 支持用户名模糊搜索。
2. 支持资料昵称模糊搜索。
3. 输入为正整数时支持用户 ID 精确搜索。
4. 补充测试。

### 7.2 [T27：补齐用户资料自动初始化](https://github.com/HeanX/iChat_Pro/issues/37)

TODO 的 T03 验收标准要求注册后自动创建资料。当前资料对象主要在进入编辑页时通过 `get_or_create` 创建。

需要：

1. 在注册成功后显式创建资料，或通过可靠的 `post_save` Signal 创建。
2. 增加注册后资料存在的测试。
3. 明确历史用户资料补齐策略。

### 7.3 [T28：补齐真实私聊发起入口](https://github.com/HeanX/iChat_Pro/issues/38)

当前已有私聊会话模型、历史接口和 WebSocket 发送，但缺少面向 UI 的“创建或获取两个联系人之间私聊会话”入口。

需要：

1. 增加 `POST /api/conversations/single/` 或等价接口。
2. 校验双方联系人关系。
3. 对相同两位用户复用已有活动私聊。
4. 增加真实会话列表接口。
5. 由 `#17` UI 使用这些接口。

### 7.4 [T29：补齐私聊联系人权限校验](https://github.com/HeanX/iChat_Pro/issues/39)

T11 文档和验收标准要求非联系人不能发送私聊消息。当前 WebSocket Consumer 主要校验双方是否为活动会话成员，未完整校验联系人关系。

需要：

1. 私聊创建时校验联系人关系。
2. 私聊发送时再次校验联系人关系或明确联系人删除后的会话策略。
3. 增加“已删除联系人不能继续发送”的测试。

### 7.5 [T30：统一协议文档与实际实现](https://github.com/HeanX/iChat_Pro/issues/40) ✅ 已完成

协议收敛已于 2026-06-03 完成。以下差异已全部对齐：

| 协议草案 | 收敛结果 |
| --- | --- |
| 私聊携带 `client_message_id` 并保持幂等 | `EncryptedMessage.client_message_id` + 唯一约束 + Consumer `IntegrityError` 保护 |
| `message.single.accepted` | Consumer 已改用 `message.single.accepted`，测试已更新 |
| `message.receipt.updated` | 统一回执协议 `message.receipt.update` → `message.receipt.updated`，测试已更新 |
| `group.members.changed` | `broadcast_group_members_changed` 已接入 invite/remove/disband 视图 |
| 群人数、文本长度、单包大小限制 | 群成员上限 50、密文解码上限 64KiB，校验已落入 `validate_private_message` 和 `validate_group_message` |

收敛交付物：

1. `client_message_id` 在私聊和群聊均强制要求（非空字符串，max 64 字符）。
2. 私聊和群聊 Consumer 均以事务内查询 + `IntegrityError` 兜底实现幂等。
3. `group.members.changed` 通过 `_broadcast_member_change` 在邀请、移除、解散群聊时推送。
4. 11 条协议级回归测试覆盖私聊和群聊的发送、转发、幂等、回执和载荷拒绝路径。

### 7.6 [T31：补齐管理员关键操作日志和软删除策略](https://github.com/HeanX/iChat_Pro/issues/41)

需求文档要求管理员管理用户、群组和消息，并记录关键操作日志。当前 Django Admin 已具备查看入口和账号启停能力，但管理员操作日志尚未形成明确实现。

1. 定义 `AdminOperationLog`。
2. 记录账号禁用或恢复、群组删除、消息删除等关键操作。
3. 明确群组和消息采用软删除还是硬删除。
4. 增加 Admin 行为测试。
5. 保持管理员无法查看消息明文，也不直接展示密文载荷。

## 8. Phase 1 验收门槛

Phase 1 完成必须同时满足以下条件。

### 8.1 用户流程

- [ ] 新用户注册后拥有资料对象和客户端身份密钥。
- [ ] 用户可以登录、退出，禁用账号无法登录。
- [ ] 用户可以按用户名、昵称或用户 ID 查找联系人。
- [ ] 用户可以发送、同意、拒绝好友申请并删除联系人。

### 8.2 私聊

- [ ] 联系人可以创建或复用真实私聊会话。
- [ ] 页面不再依赖 mock 会话或 mock 消息。
- [ ] 两位用户可以发送、接收和解密文本消息。
- [ ] 历史消息可以分页加载并解密。
- [ ] 送达、已读和错误状态可以在 UI 中展示。
- [ ] 删除联系人后的私聊发送策略已明确并有测试。

### 8.3 群聊

- [ ] 仅保留一套群聊领域模型。
- [ ] 群主可以创建群聊、修改信息、邀请和移除成员、解散群聊。
- [ ] 成员可以退出群聊。
- [ ] 外部用户和普通成员不能越权管理群成员。
- [ ] 三位用户可以实时发送、接收和解密群聊文本消息。
- [ ] 新成员无法查看加入前消息，被移除成员不再收到新消息。

### 8.4 安全

- [ ] 数据库、HTTP、WebSocket 和日志中不出现消息明文、私钥或 Session Key。
- [ ] 管理后台不展示消息明文和密文载荷。
- [ ] 密钥丢失、密钥变化和密文损坏均有明确 UI 提示。
- [ ] 私聊和群聊关键权限路径有负向测试。
- [ ] 协议版本、幂等策略和载荷限制已经固化。

### 8.5 桌面端与交付

- [ ] Electron 可以打开登录页并完成基础聊天流程。
- [ ] `python manage.py check` 通过。
- [ ] `python manage.py makemigrations --check --dry-run` 通过。
- [ ] 自动化测试通过。
- [ ] 两用户私聊和三用户群聊手动联调有记录。
- [ ] README、需求、协议、数据库、ER 图、类图和演示步骤与实际实现一致。

## 10. 审查结论

### 10.1 是否已经完成 Phase 1

否。

当前项目已经完成大量后端、安全模型和实时通信能力，自动化测试基础良好，但用户仍无法通过真实 UI 完成完整聊天闭环，Electron 也尚未交付。

### 10.2 做完当前开放 Issue 是否一定完成 Phase 1

否。

现有开放 Issue 已覆盖主要收尾工作，但至少还需要补充：

1. 联系人按昵称和用户 ID 搜索。
2. 注册后资料自动初始化。
3. 私聊创建或复用入口与真实会话列表。
4. 私聊联系人权限校验。
5. 协议文档与实现收敛。
6. 管理员操作日志与软删除策略。
7. 一期文档、ER 图和类图与实际模型对齐。

### 10.3 何时进入 Phase 2

只有在第 8 节验收门槛全部满足后，才进入 Phase 2。

Phase 2 首批建议按以下顺序实施：

```text
Channel
→ Bot 身份和命令 Bot
→ LLM Bot
→ OpenClaw Adapter
→ 受限 Agent 工具能力
```
